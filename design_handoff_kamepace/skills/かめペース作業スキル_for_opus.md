# かめペース デザイン作業スキル（Opus引き継ぎ用）

このプロジェクトでユーザーの指示を解釈・実装するときの流儀。以下は実際のやり取りから学んだ運用ルール。

## 1. 仕様の絶対原則（§1）を全変更でチェックする
- 記録率・網羅率・連続記録・達成演出は**どの画面にも絶対に置かない**
- 未記録・疲労超過に警告色や叱責コピーを使わない（100超もそのまま積むだけ）
- 疲労量の数値を直接入力させるUIを作らない。数値入力は「この枠に使った時間」の1箇所のみ
- **累積疲労の数値・ゲージ・%はホームやシャカに出さない**。累積はシャカシャカの絵文字量だけで見せる
- 「+n」表示が許されるのは記録単位・枠単位の目安まで。就寝前合計などは内部値

## 2. ユーザーの指示の解釈スタイル
- 指示は短く、UIの構造変更を含むことが多い。**大きな構造変更は作る前に設計計画を出して承認を取る**（「作る前に設計計画を教えてください」と言われた実績あり）
- 「前のに戻して」「〜と同じにして」が頻出。**過去のワイヤー・旧フロー・添付スクショが正**。勝手に改良せず、参照物のコンポーネント構造をそのまま流用する
- 添付スクショ（あすけん・Zaim等）は「このコンポーネント形式にして」という意味。色やフォントはかめペースのデザイン言語のまま、**構造だけ**参照する
- 曖昧なら質問する。仕様書と矛盾を感じたら勝手に解釈せず確認（ユーザー自身がそう指示している）
- 「今のは残して」と言われたら旧版を比較用として別セクションに残す

## 3. データモデル（会話で確定した内容）
- 階層：**大カテゴリ（9＋休憩・固定）→ 小カテゴリ（やったこと）→ 程度（ポップアップ）**
- 疲労値を持つのは小カテゴリのみ。大カテゴリは入口＋色＋アイコンの単位
- 記録フロー：記録ボタン → 大カテゴリ選択（例：家事）→ 小カテゴリ選択（例：皿洗い）→ ものによって程度ポップアップ（例：洗い物多め）
- 記録の2パターン：時間帯＞予定＞タスク ／ 時間帯＞タスク（ホームで📋予定グループと直チップで区別）
- 睡眠回復＝**就寝前の疲労 − 朝の残量申告**（時間からの計算ではない。残量から逆算）。時刻は日付を閉じる用途のみ
- 休憩＝マイナス係数の小カテゴリとして同じ列に並ぶ。後から回復（⏰）は家事・手続き系に付く内部効果
- 新規カテゴリは必ず既存のコピー＋「くらべてどう？」3択。空欄からの新規作成はさせない

## 4. デザイン言語
- フォント：Zen Kaku Gothic New（本文）／ Space Mono（数値・ラベル）／ Archivo（大数字）
- 色：ライム #c4f000（アクション・体）、ピンク #ff5fa2（心）、背景 #f7f4ec、墨 #1b1b18、白カード
- リスト行コンポーネント：裸のMaterial Symbolsアイコン（FILL 1、カテゴリ色）＋名前＋Space Monoサブテキスト＋境界線 #f1efe8。選択中は #fbfdf0 背景＋inset 3px ライム
- カート済み行：⊕ → ライム丸チェック＋行背景 #fbfdf0
- モーダルは**中央カード型**（あすけん分量ポップアップ準拠）：タイトル＋✕、対象チップ、ライムの目安バンド、セグメント、キャンセル系＋OK
- 絵文字サイズは統一（画面全体に100個収まる基準）。シャカの絵文字1個＝疲労1
- アイコンはZaim風の単色（Material Symbols Rounded）。絵文字は枠アイコン（🌅☀️🌤🌙🛏）と装飾のみ

## 5. ファイル構成と編集ルール
- `かめペース UI v1（ホーム・記録・登録確認）.dc.html` … 静的モック一式。**新しいターンのセクションは必ず最上部に追加**（turn番号＋画面ID 1a〜6c…、idバッジ＋アンカーリンク）
- `かめペース プロトタイプ.dc.html` … 触れるプロトタイプ（下部タブ常設、ホーム中央、matter.js物理シャカ）。UI一式で承認を得てから追従させる
- 各画面の下に説明カード（白枠 #e4e1d8、13px、仕様の§参照付き）を必ず置く
- 既存画面は「変えないで」と言われることが多い。**指示された画面だけを触る**
- 日本語ファイル名は run_script のパスで弾かれる。read_file/str_replace_edit を使うか、必要なら ASCII 名にコピーしてから run_script
- 同一マークアップが複数画面にある（1dと1iなど）。str_replace は前後文脈を含めて一意にする

## 6. 実装の注意
- インラインスタイルのみ。フレックス行には gap、右端要素に flex:0 0 auto + white-space:nowrap（折り返し事故が複数回あった）
- 絵文字パイルは logic class の makePile(count, seed) で生成し sc-for で描画
- プロトタイプの物理はうまくいっている参照実装（添付されたhtml）のパラメータを丸ごと移植する
- 大きな変更後は ready_for_verification、軽微なら skip_verifier_agent: true
- **残量申告は「消す量」ではなく「残す量」を選ばせる**（就寝前67の絵文字を全画面に敷き、ラインより下＝残る。回復はライン上の絵文字が🌙の雨で消える演出で見せる）
- 全画面ドラッグUIは透明レイヤー（z2, touch-action:none, pointerdown/move で clientY→値換算）を敷き、ヘッダー/CTAはその上（z4）、装飾はpointer-events:none
- 演出は @keyframes を helmet に足し、状態フラグ（sleepAnim等）で sc-if 切替 → setTimeout で画面遷移。連打ガード（if (anim) return）を忘れない

## 8. ハマりどころ（実体験）
- **str_replace_edit でメソッド定義を置換するとき、シグネチャ行（`pileGlyphs() {`）を old/new 両方に含めると重複行が生まれる**。実際 `pileGlyphs() {` が2行できてDCの「logic class eval FAILED（本文はpropsのみ描画＝画面が真っ白）」に。→ メソッドを書き換えるときは中身だけを置換し、シグネチャ行は片方だけにする
- **DCの logic 破損は get_webview_logs に "logic class eval FAILED" で出る／画面はnavだけ描画され本文が空**。この症状を見たら即JS構文を疑う
- **構文エラーの特定**：Japanese名はrun_script不可 → ASCIIにcopy_files → `data-dc-script>([\s\S]*?)</script>` を抜いて `new Function(body)` で試す（sandboxのエンジンは行番号がズレるので、行番号より「重複/未閉じ」をgrepで探す方が速い）
- 物理ループが回っている画面では eval_js の多段 await が10秒タイムアウトしがち。状態確認は短く、または get_webview_logs で代替
